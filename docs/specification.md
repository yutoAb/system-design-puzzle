# システム設計パズル 仕様

## 目的

システム設計パズルは、システム設計面接で問われるトレードオフを練習するブラウザゲームです。プレイヤーはお題を読み、アーキテクチャのピースを選び、可用性、低レイテンシ、コスト、要件充足度、スケール性の観点でフィードバックを受けます。

## MVPの範囲

最初のプレイ可能版では、次のお題を扱います。

- 100万人が同時視聴する動画配信サービスを設計する。

プレイヤーは固定のピース一覧から、必要なピースをボードへドラッグして配置します。

- CDN
- ロードバランサー
- アプリケーションサーバー
- オブジェクトストレージ
- リレーショナルDB
- DBレプリカ
- キャッシュ
- メッセージキュー
- レートリミッター
- コンシステントハッシュ
- キーバリューストア
- 分析パイプライン

提出された設計では、ボードに配置済みのピースだけを採点対象にします。アプリケーションは次を返します。

- `totalScore`: 0から100までの整数
- `dimensionScores`: スケール性、可用性、低レイテンシ、コスト効率、要件充足
- `strengths`: 設計の強み
- `risks`: 不足または弱い点
- `interviewNotes`: 面接で議論すべき観点

## 採点ルール

採点は決定的です。同じお題と同じピース集合なら、常に同じ結果を返します。

動画配信のお題では、次の考え方で採点します。

- CDNは、地域をまたいだ低レイテンシ配信に重要。
- オブジェクトストレージは、動画ファイルの保存に重要。
- ロードバランサーとアプリケーションサーバーは、リクエスト処理の中核。
- キャッシュは、読み取り負荷と応答時間を減らす。
- メッセージキューは、非同期処理と急な負荷増加に効く。
- リレーショナルDBは、メタデータ管理に有効だがコストが増える。
- DBレプリカは、読み取り性能と可用性の論点になる。
- レートリミッターは、乱用や急な書き込み増加への対策になる。
- コンシステントハッシュは、キャッシュや保存先の分散を説明しやすい。
- キーバリューストアは、大規模な単純参照に向いている。
- 分析パイプラインは、中核配信には必須ではないが改善サイクルに役立つ。

総合スコアは、次の平均を丸めて計算します。

- スケール性
- 可用性
- 低レイテンシ
- コスト効率
- 要件充足

## 参考資料から取り入れたゲーム性

参考資料では、システム設計は唯一の正解を当てるものではなく、要件、概算、ボトルネック、トレードオフを議論するものとして扱われています。ゲームではそれを次の流れに変換します。

1. 機能要件と非機能要件を読む。
2. 限られたアーキテクチャ予算でピースを選ぶ。
3. ピースをエッジ、流量制御、サービス、保存、非同期、データ分散のレーンへ配置する。
4. まず中核の処理経路を成立させる。
5. 任意のピースでスケール性、信頼性、面接での議論カバレッジを高める。
6. 採点後、面接官から聞かれそうな観点を確認する。

## クリーンアーキテクチャ

依存方向は内側へ向けます。

- `domain`: 採点ルール、チャレンジ、コンポーネント定義。
- `application`: ユースケース。
- `infrastructure`: インメモリリポジトリなどの外部実装。
- `interface-adapters`: UIやサーバーとユースケースの変換層。
- `public`: ブラウザUI。

ドメイン層はUI、サーバー、保存方式に依存しません。

## 受け入れ条件

- プレイヤーは利用可能なお題を取得できる。
- プレイヤーは利用可能な設計ピースを取得できる。
- プレイヤーはお題に対して配置済みピースを提出できる。
- システムは決定的な採点結果とフィードバックを返す。
- 中核挙動はUI実装より先に自動テストで確認されている。
